# System-Next Improvement Review

## Purpose

This review locks the next evidence requirements for the system-level improvement chain. Each challenge must be answered by tracked tests, runtime artifacts, or generated evaluation summaries before the claim can be promoted into README, thesis, or operations wording.

## Challenge To Evidence Map

| Challenge | Required Evidence | Proof Location | Acceptance Rule |
| --- | --- | --- | --- |
| Algorithms are registered tools, not free-form LLM calls | `ToolSpec` registry tests and executor dispatch tests | unit tests for registry shape, handler binding, input/output schema enforcement, and executor lookup | Every executable algorithm referenced by a plan resolves through a registered `ToolSpec`; executor dispatch rejects unknown or schema-invalid tools instead of relying on free-form LLM output. |
| LLM planning is KG-grounded | Per-step grounding report in `plan.json`, `audit.jsonl`, and inspection output | run artifacts plus `GET /api/v2/runs/{run_id}/inspection` | Every planned step records KG evidence such as selected task, data type, source, algorithm, constraint, or validator reference; missing evidence is explicit rather than silent. |
| KG improves accuracy | Ablation JSON/Markdown with validity and success metrics generated by `scripts/eval_kg_ablation.py` from supplied rows only | generated ablation summary under `docs/superpowers/specs/` | Compare full KG-grounded planning against constrained baselines and report planning validity, unknown-algorithm rate, execution success, grounding score, and case count without inventing live metrics. |
| Unsupported requests are controlled | API/service guard tests for GDP, Chinese schema rename, and off-domain requests | API and service tests that exercise unsupported or out-of-scope requests | The runtime returns controlled rejection or clarification responses with machine-readable reasons; it must not synthesize unsupported workflows or silently map them onto unrelated handlers. |
| Long-running runs are observable | Planning telemetry, step-level audit events, and checkpoint metadata | `audit.jsonl`, `run.json`, `plan.json`, and inspection output | Operators can see planning start/end, step lifecycle, checkpoint timestamps, current phase, and last progress marker without requiring a UI. |
| Recovery is bounded and honest | Stale-run scanner returns explicit recovery actions without claiming full 7x24 HA | scanner/service tests and operations documentation | Stale runs are classified into concrete actions such as resume, fail, retry checkpoint, or manual intervention; documentation must not claim full 7x24 high availability. |

## Claim Lock

The next implementation may strengthen the existing no-UI runtime claim only through the evidence above. It must not widen the stable runtime surface beyond the current task-driven building, road, water, and bounded POI slices, and it must keep trajectory-to-road as a reserved seam until executable runtime evidence exists.
