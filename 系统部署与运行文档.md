# FusionAgent 系统部署与运行文档

## 1. 文档范围

本文档只描述当前仓库中“已经真实实现”的本机可用 MVP 部署方式，不覆盖研究目标的全部终态能力。
当前主要目标是：

- 在你的电脑上稳定运行
- 能处理 `building` / `road` 两类 SHP 融合任务
- 能通过 API 查看 run 状态、plan 和 artifact

## 2. 环境要求

### 操作系统

- Windows 10/11
- Linux
- macOS

### Python

- 推荐 `3.9 - 3.11`

### 外部依赖

- Redis
- Neo4j 5.x

### Python 依赖安装

```bash
python -m pip install -r requirements.txt
```

## 3. 配置方式

### 3.1 `.env.example`

项目根目录已经提供 [.env.example](/E:/vscode/fusionAgent/.env.example)，可作为环境变量模板。

### 3.2 本地私有配置文件

如果使用本地直跑脚本，推荐复制 [依赖.txt.example](/E:/vscode/fusionAgent/依赖.txt.example) 为 `依赖.txt`，由脚本自动读取：

```bash

```

### 3.3 关键环境变量

#### 核心

- `GEOFUSION_KG_BACKEND=memory|neo4j`
- `GEOFUSION_LLM_PROVIDER=mock|openai|auto`
- `GEOFUSION_CELERY_EAGER=1|0`

#### Neo4j

- `GEOFUSION_NEO4J_URI`
- `GEOFUSION_NEO4J_USER`
- `GEOFUSION_NEO4J_PASSWORD`
- `GEOFUSION_NEO4J_DATABASE`

#### LLM

- `GEOFUSION_LLM_BASE_URL`
- `GEOFUSION_LLM_MODEL`
- `OPENAI_API_KEY` 或 `GEOFUSION_LLM_API_KEY`

## 4. Neo4j 隔离说明

### 4.1 Enterprise

如果是 Neo4j Enterprise，可以通过 `GEOFUSION_NEO4J_DATABASE` 指向 FusionAgent 专用数据库。

### 4.2 Community

我已经按当前仓库实现把 Community 作为默认兼容目标处理。
Community 不支持多个用户数据库时，FusionAgent 会自动探测当前 home database，并通过 `:FusionAgentManaged` 标签隔离自己的子图。

这意味着：

- 不会误查你现有的无关业务节点
- 只重置 FusionAgent 自己管理的节点
- 不要求你先删掉其他业务图数据

### 4.3 图状态检查

```bash
python scripts/inspect_neo4j_state.py
python scripts/inspect_neo4j_state.py --managed-only
```

## 5. 启动方式

### 5.1 本地快速模式

适合调试 API 和跑测试，不依赖真实 Neo4j / Redis / 在线模型。

```bash
set GEOFUSION_KG_BACKEND=memory
set GEOFUSION_LLM_PROVIDER=mock
set GEOFUSION_CELERY_EAGER=1
uvicorn main:app --reload --host 0.0.0.0 --port 8000
```

### 5.2 本地全链路模式

适合联调 Neo4j、Redis、Celery 和真实 LLM provider。

```bash
python scripts/start_local.py --check-only
python scripts/start_local.py
```

如果需要只重建 FusionAgent 自己管理的图：

```bash
python scripts/start_local.py --check-only --reset-managed-graph
```

### 5.3 Docker Compose

```bash
copy .env.example .env
docker compose up --build
```

默认会启动：

- `api`
- `worker`
- `scheduler`
- `redis`
- `neo4j`

## 6. API 访问

### 6.1 文档入口

- Swagger UI: `http://127.0.0.1:8000/docs`
- ReDoc: `http://127.0.0.1:8000/redoc`

### 6.2 `v2` 真实请求方式

`POST /api/v2/runs` 当前真实接口是 `multipart/form-data` 上传，不是 JSON 直接提交。

示例：

```bash
curl -X POST "http://127.0.0.1:8000/api/v2/runs" ^
  -F "osm_zip=@tests/golden_cases/building_disaster_flood/input/osm.zip" ^
  -F "ref_zip=@tests/golden_cases/building_disaster_flood/input/ref.zip" ^
  -F "job_type=building" ^
  -F "trigger_type=disaster_event" ^
  -F "trigger_content=flood building fusion" ^
  -F "disaster_type=flood" ^
  -F "field_mapping={}" ^
  -F "target_crs=EPSG:32643"
```

### 6.3 当前主要接口

#### v1

- `POST /api/v1/fusion/building/jobs`
- `POST /api/v1/fusion/road/jobs`
- `GET /api/v1/jobs/{job_id}`
- `GET /api/v1/jobs/{job_id}/artifact`

#### v2

- `POST /api/v2/runs`
- `GET /api/v2/runs/{run_id}`
- `GET /api/v2/runs/{run_id}/plan`
- `GET /api/v2/runs/{run_id}/artifact`

## 8. 当前边界与非目标

当前仓库已经具备初步应用价值，但请按以下边界理解：

- 这是领域受限的 agentic workflow 原型
- 当前没有完整前端 Web 产品
- 当前没有真正的 `replan`
- 当前没有完整经验写回学习闭环
- 当前重点是“本机可用 MVP”，不是最终服务器生产版
