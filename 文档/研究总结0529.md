## 1. 整体研究思路与研究方向

### 1.1 研究问题的核心转变

传统 GIS 数据处理系统通常依赖人工选择数据源、配置算法参数、组织处理流程并检查结果。近年来的 GeoAI 或 geospatial agent 研究尝试用大模型理解用户意图、调用 GIS 工具或生成空间分析代码，但这类系统往往存在三个问题：

1. 任务规划与真实可执行算法之间缺少稳定约束，容易生成不可执行或语义漂移的处理链。
2. 数据融合任务涉及数据源、格式、坐标系、字段映射、空间裁剪、参数阈值和输出契约，单纯依赖自然语言推理难以保证一致性。
3. 运行结果如果只给最终文件，缺少过程证据、失败修复记录和可复现实验资产，难以支撑科研评估和工程审计。

FusionAgent 的研究思路正是围绕这些问题展开：不把智能体定义为开放式的“大模型自由工具调用器”，而把它定义为一个受知识图谱和运行时契约约束的地理融合工作流智能体。系统的智能性不主要体现在无限开放的行动空间，而体现在有限任务域内的规划、约束、验证、修复、记忆和证据化。

### 1.2 核心方法论：算法 - 任务 - 数据三元结构

开题阶段曾使用“灾害 - 数据 - 算法”作为主线。经过工程实现后，当前更合理的论文表达是：

- 可执行核心：`Algorithm - Task - Data`
- 场景约束层：`DisasterEvent / ScenarioProfile / DataNeed / OutputRequirement / QoSPolicy`

这样调整的原因是，灾害类型本身通常并不直接决定底层算法，真正决定工作流能否执行的是任务、数据和算法之间的匹配关系。例如洪水、地震、台风都可能需要建筑物、道路、水系数据，但它们对任务优先级、数据源偏好、输出字段和质量要求有所不同。因此，灾害更适合放在场景约束层，用于影响：

- 任务激活
- 数据需求与数据获取策略
- 参数策略
- 输出要求
- 评价与可靠性策略

论文中可以将这一点作为系统建模的关键思想：FusionAgent 不是简单把灾害类型映射到固定算法，而是将灾害语义转译为可执行任务束和运行约束。

### 1.3 智能体范式：受约束 Plan-and-Execute 与反应式修复

FusionAgent 当前最适合被定义为：

> Constrained Plan-and-Execute with Reactive Healing

其含义是：

- 主链路采用先规划后执行，而不是连续对话式 ReAct。
- 规划阶段由知识图谱、任务类型、数据类型、算法原语、参数规格和输出契约约束。
- LLM 可以参与意图理解和计划补全，但不能绕过运行时验证。
- 执行失败后，系统通过 repair、fallback、replan 和 checkpoint recovery 进行有限修复。
- 每次运行都保留审计记录和证据文件。

这种范式的思想是：在高风险、强约束的地理空间数据处理任务中，可靠性不应依赖模型“看起来会推理”，而应依赖显式的结构化约束、状态机、验证器、错误分类和证据写回。

### 1.4 论文主线建议

当前论文可以围绕三个主研究问题组织：

| 研究问题 | 核心关注 | 对应系统能力 |
| --- | --- | --- |
| RQ1 | 知识图谱约束是否提升地理融合工作流规划与执行有效性 | KG-grounded planning、workflow validation、algorithm/data/task binding |
| RQ2 | 反应式修复与重规划是否提升系统鲁棒性 | repair、fallback、replan、checkpoint recovery |
| RQ3 | 运行证据契约是否提升可审计性、可复现性和论文实验组织能力 | run evidence、audit、report、freeze、inspection API |

建筑物、道路、水系和 POI 融合可以作为系统能力和任务覆盖面的验证对象；自动下载、大范围拼接裁剪、报告、错误恢复和有界学习则作为支撑系统自治性、鲁棒性和研究价值的机制。

## 2. 总体系统架构

### 2.1 双入口运行模式

系统支持两类入口：

1. `task-driven`：用户直接提出具体数据融合任务，例如“融合某区域建筑物数据”“生成某城市道路融合结果”。
2. `scenario-driven`：从灾害事件或本地触发文件出发，生成一个场景级任务集合，再创建多个子任务运行。

二者共享后续运行链路：

```text
Input
-> Intent / Scenario Resolution
-> TaskBundle
-> KG Retrieval
-> Planning
-> Validation
-> Input Acquisition
-> Source Semantic Binding
-> Execution
-> Healing / Replan
-> Writeback
-> Report / Evidence Freeze
```

### 2.2 核心模块分工

| 模块 | 作用 |
| --- | --- |
| `agent/planner.py` | 根据任务、图谱上下文和运行约束生成工作流计划 |
| `agent/validator.py` | 对算法、输入输出、参数、数据类型和 schema policy 做验证与修复 |
| `agent/executor.py` | 执行工作流步骤，调用融合算法或适配器 |
| `services/agent_run_service.py` | 单次 run 的总控服务，串联规划、验证、获取、执行、修复、写回、报告 |
| `services/input_acquisition_service.py` | 根据 source id、缓存和 request bbox 准备输入 bundle |
| `services/source_asset_service.py` | 负责官方或本地数据源 materialization、下载、缓存、裁剪 |
| `services/large_area_runtime_service.py` | 统一大范围 tile、clip、per-tile fusion、stitch、dedupe、evidence 输出 |
| `services/domain_fusion_runners.py` | 将 road、water、waterways、POI、building 等领域算法包装成统一 tile runner |
| `services/run_report_service.py` | 生成 run 级报告、过程评价、结果评价、自学习边界说明 |
| `worker/tasks.py` 与 `worker/celery_app.py` | 提供 scheduled tick、recovery tick、异步任务和长时间运行基础 |

### 2.3 证据契约

FusionAgent 的一个重要研究贡献是把运行过程变成可审计证据。单次 run 的核心证据包括：

- `run.json`
- `plan.json`
- `validation.json`
- `audit.jsonl`
- artifact bundle
- `source_semantic_contract.json`
- `tile_manifest.json`
- `selected_sources.json`
- `stitched_artifact.json`
- `fusion_stats.json`
- `documents/run_report_summary.json`
- `documents/run_report.zh.md`
- `documents/run_report.en.md`

这些文件可以服务于三类用途：工程排查、运行复现和论文实验冻结。

## 3. 功能 1：无人值守自动融合与长时间运行

### 3.1 功能描述

无人值守能力指系统在接收自然语言任务、定时任务或本地场景事件后，能够自动完成任务创建、规划、验证、数据准备、融合执行、结果写回和报告生成，并在可恢复失败场景下尽量自动重试或从 checkpoint 继续执行。

当前支持的触发方式包括：

- API 直接创建 run。
- `task-driven_auto` 方式由用户描述任务，系统自动选择 source 和 workflow。
- 本地 scenario inbox：通过 `scripts/watch_scenario_inbox.py` 读取事件 JSON，创建场景运行。
- scheduled tick：通过 `GEOFUSION_SCHEDULED_RUNS` 配置定时触发任务。
- recovery tick：周期性扫描 stale run 并尝试 checkpoint redispatch。

### 3.2 能力边界

当前能力属于“no-UI mature agent”级别，不是生产级 7x24 高可用系统。其边界包括：

- 不声明外部实时灾害事件流接入。
- 不声明生产级多租户、鉴权、监控、弹性扩缩容和 SLA。
- 不声明任意任务族自动扩展，只支持 building、road、water、bounded POI。
- AOI geocoding 和远程数据源下载可能依赖外部服务可用性。
- 长时间运行主要依靠 worker、beat、checkpoint 和 recovery，不等同于完整分布式容错平台。

### 3.3 底层原理

无人值守运行的基础不是让大模型持续在线决策，而是将任务生命周期拆成显式状态机：

```text
queued -> planning -> validation -> execution -> writeback -> succeeded / failed
```

每个阶段都将状态写入 `run.json`，并将事件追加到 `audit.jsonl`。系统通过 checkpoint 记录当前阶段、计划版本、步骤位置和尝试次数。当运行停滞或失败后，recovery service 可以根据 phase、checkpoint 和 failure details 判断恢复动作，例如重新分发运行、从 checkpoint 恢复或提示人工介入。

### 3.4 技术工程实现

核心实现位于 `services/agent_run_service.py`：

- `create_run()` 创建 run 目录、请求文件、运行状态和 runtime snapshot。
- `execute_run()` 串联 planning、validation、input acquisition、execution、writeback、report。
- `_checkpoint()` 统一生成 checkpoint payload。
- `collect_recoverable_runs()` 调用 recovery service 扫描可恢复运行。
- `resume_run_from_checkpoint()` 根据恢复动作重新执行。

异步和周期任务位于 `worker/tasks.py` 与 `worker/celery_app.py`：

- `geofusion.scheduled_tick` 根据环境变量中的配置创建 scheduled run。
- `geofusion.recovery_tick` 周期扫描 stale run。
- `geofusion.recover_run` 执行单个 run 的恢复。
- Celery beat 中注册 `scheduled-run-producer` 和 `recovery-run-producer`。

场景触发实现位于 `scripts/watch_scenario_inbox.py`，支持把成功事件移动到 processed 目录，把非法或失败事件移动到 failed 目录。

### 3.5 设计初衷与思想

该功能的设计思想是将“无人值守”收敛为可验证的自动化运行闭环，而不是追求开放式自主行动。论文中可以强调：无人值守能力的可靠性来自阶段化状态、显式 checkpoint、审计日志、错误分类和有界恢复策略，而非依赖模型持续猜测下一步。

## 4. 功能 2：建筑物矢量融合与高度栅格参与融合

### 4.1 功能描述

建筑物融合是 FusionAgent 的核心任务之一。当前能力包括：

- 支持 OSM building 与 Microsoft building 等矢量建筑物数据融合。
- 支持多源 building source 的优先级融合。
- 支持高度栅格参与建筑物属性补全，生成 `height_raster`、`height_final`、`height_final_source` 等字段。
- 支持大范围 tile 化处理和最终拼接。
- 支持 source semantic contract 记录矢量源、栅格源和字段语义。

### 4.2 能力边界

当前 2026-05-28 证据冻结中，建筑物矢量 + 可选高度栅格已经作为共享运行时支持能力。但边界仍包括：

- 默认自动化 building bundle 主要是 `raw.osm.building + raw.microsoft.building`。
- Google/OpenBuildingMap/Google Open Buildings Vector/local Microsoft 仍属于 manual-preload supplement，不能宣称全自动远程可得。
- 高度栅格参与融合的前提是 source semantic contract 能解析到可读取 raster path。
- 规模化真实数据验证仍需要远端 provider 可用或本地缓存存在。

### 4.3 底层原理

建筑物融合的核心问题是多源 footprint 之间存在位置偏差、尺度差异、重复覆盖、缺失和属性不一致。当前系统采用分解式思想处理：

1. source normalize：统一源数据字段、CRS 和几何有效性。
2. candidate graph：构造候选匹配关系。
3. component solver：判断一对一、一对多、多对多关系。
4. cascade geometry priority：根据源优先级和匹配关系选择几何。
5. residual priority：保留未匹配但可信的残差建筑。
6. conflict optimization：处理重叠、道路冲突等几何问题。
7. height from raster：从高度栅格抽取或补全建筑高度。
8. quality metrics：输出质量指标和统计。

从论文角度看，建筑物融合可作为“复杂融合任务”的代表，因为它同时涉及几何匹配、属性融合、栅格辅助、冲突优化和大范围 tile 化。

### 4.4 技术工程实现

主要实现组件包括：

- `fusion_algorithms/building_matching_v8.py`
- `fusion_algorithms/building_height.py`
- `fusion_algorithms/building_raster.py`
- `fusion_algorithms/building_optimization.py`
- `services/tiled_building_runtime_service.py`
- `services/domain_fusion_runners.py` 中的 `make_building_multisource_runner()`
- `services/agent_run_service.py` 中的 `run_multisource_building_execution_stage()`
- `services/source_semantic_contract_service.py`

在共享大范围运行时中，建筑物融合被包装成 tile runner。每个 tile 对源数据做 buffered clip，执行 tile 内融合，再通过 owner bbox stitch 避免 tile 边界重复。栅格源通过 `raster_sources` 传入建筑物 tiled runtime，用于高度字段抽取和最终高度字段决策。

### 4.5 功能设计初衷与思想

建筑物融合功能的设计初衷是验证 FusionAgent 不只是简单调度单个脚本，而能处理多源、多尺度、多模态的真实空间融合任务。高度栅格的引入进一步体现了系统对“矢量 + 栅格”联合融合的支持，使论文方法可以覆盖更复杂的数据融合范式。

## 5. 功能 3：道路矢量融合

### 5.1 功能描述

道路融合支持 OSM road 与 Overture transportation 等道路/交通数据源之间的矢量融合。当前道路能力已经进入共享大范围运行时，支持：

- 自动或缓存方式获取 OSM road 与 Overture transportation。
- 对道路线数据进行 tile clip。
- 使用 V7 road conflation 算法进行匹配、残差保留和去重。
- 输出 fused road artifact、统计信息和 source lineage。

### 5.2 能力边界

当前道路矢量融合已经属于稳定支持能力，但边界包括：

- Overture 数据下载依赖 `overturemaps` 或 `uvx overturemaps`，也可使用显式 URL 或本地缓存。
- trajectory-to-road 仍是 reservation-only，不属于当前道路融合能力。
- 当前任务重点是道路矢量数据源之间的 conflation，不是轨迹推断、路网拓扑修复或交通仿真。

### 5.3 底层原理

道路融合的核心问题是线状目标匹配。不同来源的道路可能存在：

- 几何偏移
- 分段粒度不同
- 道路等级字段不一致
- 局部缺失或重复
- 交叉、回折和残差线段

V7 road conflation 的基本思想是：

1. 将长线段按角度和长度规则切分。
2. 在空间缓冲区内寻找候选匹配。
3. 使用 Hausdorff 距离、方向角、长度相似度和覆盖比例判断匹配。
4. 对匹配线段优先保留 base source，同时保留 supplement 中未覆盖的残差。
5. 对重复线段、近 base 回流线和交叉残差做清理。

### 5.4 技术工程实现

主要实现包括：

- `fusion_algorithms/road_conflation_v7.py`
- `fusion_algorithms/line_conflation_v7.py`
- `services/domain_fusion_runners.py` 中的 `run_road_tile()`
- `services/large_area_runtime_service.py`
- `services/source_asset_service.py` 中的 Geofabrik 和 Overture materialization

在 `run_road_tile()` 中，系统将 `raw.osm.road` 作为 base，将 `raw.overture.transportation` 或 `raw.overture.road` 作为 supplement，读取后统一 CRS，调用 `run_road_conflation_v7()`，并输出算法 id、统计、配置和 warnings。

### 5.5 功能设计初衷与思想

道路融合用于证明 FusionAgent 的共享运行时不仅适用于面状建筑物，也适用于线状网络对象。它展示了系统可以在同一规划、验证、下载、tile、stitch 和报告框架下接入不同几何族算法。

## 6. 功能 4：水系矢量融合，包括河流和湖泊

### 6.1 功能描述

水系融合覆盖两类对象：

- 面状水体：湖泊、水库、宽河面等 water polygon。
- 线状水系：河流、水道、水网等 waterways line。

当前共享运行时中，`JobType.water` 可以内部拆成两个 slice：

- `water_polygon`：OSM water + HydroLAKES 或 local water。
- `waterways_line`：OSM waterways + HydroRIVERS 或 local waterways。

输出中通过 `feature_kind=polygon|line` 保留对象类型。

### 6.2 能力边界

当前水系融合已经稳定支持，但边界包括：

- HydroLAKES/HydroRIVERS 远程数据源下载依赖外部 provider 可用性。
- 水系融合聚焦矢量对象对齐与补充，不包括水动力模拟、洪水淹没模拟或实时水文预报。
- 河流线和湖泊面在内部是不同 slice，外部统一为 water 任务。

### 6.3 底层原理

水系融合采用“面水体优先合并 + 线状水系 conflation”的组合策略。

面水体融合的核心是 polygon priority merge：当 OSM water 与 HydroLAKES 等参考源存在重叠时，按照覆盖、优先级和 source lineage 合并；当补充源中存在 OSM 不覆盖的区域时，保留为残差面。

线状水系融合与道路融合相似，但参数更适合河流/水道：允许更大的自然弯曲、不同的匹配缓冲区和残差长度阈值。其目标是在保持主河道几何可信度的同时补充缺失支流和局部水道。

### 6.4 技术工程实现

主要实现包括：

- `fusion_algorithms/water_fusion.py`
- `fusion_algorithms/waterways_conflation_v7.py`
- `adapters/water_adapter.py`
- `services/domain_fusion_runners.py` 中的 `run_water_polygon_tile()` 和 `run_waterways_tile()`
- `services/source_asset_service.py` 中的 HydroSHEDS materialization

在共享大范围运行时中，两个 water slice 可以分别执行 tile 内融合，再在最终 stitched artifact 中合并。`LargeAreaRuntimeService` 本身不关心水系算法细节，只负责 tile 管理、输入裁剪、输出拼接、去重和证据文件。

### 6.5 功能设计初衷与思想

水系融合的设计价值在于展示系统对同一任务类型内部异质对象的处理能力。水系既有 polygon 又有 line，这要求运行时不仅能按 job_type 选择算法，还要能按 geometry family 拆分子流程。这对论文中“可扩展但有界的融合工作流编排”是重要证据。

## 7. 功能 5：POI 数据融合，包括 OSM 和 GeoNames/GNS

### 7.1 功能描述

POI 融合支持在有界 AOI 内融合 OSM POI 与 GNS/GeoNames gazetteer 数据。当前能力包括：

- 自动或缓存方式获取 OSM POI 和 GNS/GeoNames。
- 通过 geohash 邻域和名称相似度进行候选匹配。
- 根据 source priority 处理重复和冲突。
- 输出 canonical id、canonical name、canonical category、source id、source rank、MATCHED 等字段。

### 7.2 能力边界

POI 是有界支持，不是通用实体对齐系统。边界包括：

- 必须有 AOI 或任务边界。
- 不支持“全局所有 POI 实体对齐”。
- 不宣称解决复杂商户实体合并、品牌知识库融合、多语言别名库和动态营业状态。
- `raw.rh.poi` 仍属于 manual-preload supplement。

### 7.3 底层原理

POI 融合的核心是点目标实体匹配。与建筑物或道路不同，POI 的几何位置通常只是候选依据之一，名称和类别语义同样重要。当前采用 geohash + name similarity 的轻量融合策略：

1. 将点目标映射到 geohash 网格。
2. 在邻域 ring 内寻找候选点。
3. 比较名称相似度与类别信息。
4. 对匹配实体生成 canonical 字段。
5. 对未匹配实体按 source priority 保留。

这种策略适合 bounded AOI 内的灾害响应数据准备，能在不引入复杂实体知识库的情况下完成实用级 POI 合并。

### 7.4 技术工程实现

主要实现包括：

- `fusion_algorithms/poi_fusion.py`
- `adapters/poi_adapter.py`
- `services/domain_fusion_runners.py` 中的 `run_poi_tile()`
- `services/runtime_source_aliases.py` 中的 GNS/GeoNames alias
- `services/source_asset_service.py` 中的 GNS country file 解析与缓存

`run_poi_tile()` 会将 `raw.gns.poi` 和 `raw.geonames.poi` 归并到 GNS 语义下，按 `POI_SOURCE_PRIORITY_ORDER` 组织数据源，补齐 `canonical_id`、`canonical_name` 和 `canonical_category`，再调用 geohash priority fusion。

### 7.5 功能设计初衷与思想

POI 功能的意义在于补齐灾害响应中的“设施、地名、兴趣点”数据层。它也用于展示系统对点、线、面三类几何对象的统一融合框架。但为了避免论文过度宣称，当前应始终强调它是 bounded POI fusion，而不是开放域实体解析系统。

## 8. 功能 6：大范围数据融合，包括拼接和裁剪

### 8.1 功能描述

大范围数据融合是第 2-5 项能力从“小 AOI 或单文件处理”提升到区域级运行的基础。当前共享大范围运行时支持：

- 根据 AOI bbox 生成 tile manifest。
- 对每个 tile 使用 buffered bbox 裁剪输入源。
- 每个 tile 独立调用领域融合 runner。
- 按 owner bbox 拼接 tile 输出，避免边界重复。
- 通过 canonical id、source feature id 或 geometry WKB 去重。
- 可选地按最终边界裁剪。
- 输出 tile、source、stitch、stats 和 artifact metrics 证据。

### 8.2 能力边界

当前大范围运行时已经支持 building、road、water、POI，但边界包括：

- 当前实现偏工程研究型运行时，不是分布式 Spark/GPU 级大数据处理平台。
- `LargeAreaRuntimeService` 当前以顺序或有限 worker 方式执行，仍需要更强性能评估。
- 远程源 provider 不可用时，live-source smoke 可以被记录为 provider-specific skip。
- 对极大国家级数据的内存、并发、磁盘和失败恢复仍需更多 soak test。

### 8.3 底层原理

大范围融合的关键问题是直接全量加载与全局匹配成本高，且边界处容易产生重复或断裂。当前采用 tile-buffer-owner 机制：

1. Tile partition：将 AOI 分成规则 tile。
2. Buffered input：每个 tile 读取比 owner bbox 更大的 buffered bbox，保证边界附近匹配充分。
3. Per-tile fusion：tile 内执行领域算法。
4. Owner bbox stitch：最终只保留属于 tile owner bbox 的结果，减少重叠重复。
5. Dedupe：通过 canonical id 或 WKB 去重。
6. Final clip：可按 AOI 或国家边界裁剪。

### 8.4 技术工程实现

主要实现位于 `services/large_area_runtime_service.py`：

- `LargeAreaSlice` 描述一个领域 slice，包括 name、geometry_family、sources、runner 和 parameters。
- `LargeAreaRuntimeService.run()` 负责任务总控。
- `_clip_sources_for_tile()` 负责 tile 输入裁剪。
- `_stitch()` 负责拼接、owner bbox 筛选、去重和最终裁剪。
- `_dedupe()` 先按 canonical/source id 去重，再按 geometry WKB 去重。
- 输出 `tile_manifest.json`、`selected_sources.json`、`stitched_artifact.json`、`fusion_stats.json`。

`services/domain_fusion_runners.py` 将不同算法统一包装成 `DomainRunner`，使大范围运行时不需要关心道路、水系、POI 或建筑物的算法细节。

### 8.5 功能设计初衷与思想

大范围运行时的设计目标是把“规模化处理”从单个算法中剥离出来，形成跨领域复用的基础设施。这样可以避免每个领域重复实现 tile、clip、stitch 和 evidence 逻辑，也让论文能够把第 2-5 项能力统一归纳为“共享运行时上的多几何族融合”。

## 9. 功能 7：自动数据下载

### 9.1 功能描述

自动数据下载指系统根据任务和 source id 自动解析、获取、缓存、裁剪并准备融合所需数据源。当前支持的数据源类型包括：

- OSM/Geofabrik shapefile bundle。
- Microsoft building footprints。
- Overture transportation。
- HydroRIVERS 与 HydroLAKES。
- GNS/GeoNames gazetteer。
- repo-local manual-preload source。

### 9.2 能力边界

当前自动下载是 bounded official-source materialization，不是任意互联网数据采集。边界包括：

- 不支持所有国家、所有源、所有格式的任意自动下载。
- provider URL、API、命令行工具和网络状态会影响成功率。
- 部分 Google/OpenBuildingMap/local source 仍需 manual preload。
- Overture 下载依赖本机安装 `overturemaps` 或 `uvx` fallback。
- 自动下载失败可以被识别和报告，但不能保证所有 provider 问题自动修复。

### 9.3 底层原理

数据下载不是简单 HTTP fetch，而是 source id 驱动的 materialization：

```text
source_id + AOI
-> source catalog / local bundle / provider resolver
-> remote or local asset
-> cache
-> vector loading
-> bbox clip
-> feature count / coverage
-> component coverage
-> input bundle
```

核心思想是将数据源可得性、缓存命中、裁剪结果、coverage empty 和 fallback 都显式记录，避免运行时在数据缺失时静默失败。

### 9.4 技术工程实现

主要实现包括：

- `services/source_asset_service.py`
- `services/input_acquisition_service.py`
- `services/local_bundle_catalog.py`
- `scripts/materialize_source_assets.py`

`SourceAssetService.resolve_raw_source_path()` 根据 source id 分派到不同 resolver：

- `_resolve_geofabrik_source()`
- `_resolve_msft_buildings()`
- `_resolve_overture_transportation()`
- `_resolve_hydrosheds_water()`
- `_resolve_gns_poi()`
- repo-local source resolution

`InputAcquisitionService` 则负责把 materialized source 组织成 run 可消费的 input bundle，并记录 `source_mode`、`cache_hit`、`selected_source_id` 和 `component_coverage`。

### 9.5 功能设计初衷与思想

自动下载的目标是减少人工准备数据的成本，让系统能够从“用户提出区域任务”推进到“自动准备官方源并执行融合”。但设计上没有把它包装成无边界爬虫，而是坚持 source catalog 和 source contract 驱动。论文中可以把它作为可复现性的支撑机制，而不是主要贡献本身。

## 10. 功能 8：融合任务报告生成，包括过程评估和结果评估

### 10.1 功能描述

系统在任务完成后生成运行报告，包括：

- 中文 Markdown 报告。
- 英文 Markdown 报告。
- JSON summary。
- 过程评价。
- 结果评价。
- 大范围运行时证据。
- source semantic contract。
- fallback summary。
- self-evolution boundary。

### 10.2 能力边界

当前报告已经具备工程和论文证据组织价值，但不是完整学术实验结果报告。边界包括：

- 结果评价主要包括 artifact validity、feature count、schema validation 等工程指标。
- 如果需要精度、召回率、IoU、Hausdorff 分布、人工标注对比等论文质量指标，需要额外 benchmark 数据和评测脚本。
- 报告不会替代最终论文的统计分析和图表。

### 10.3 底层原理

报告生成的基本思想是从 run 状态、plan、audit events、artifact 和 source semantic contract 中抽取证据，形成结构化 summary，再渲染为中英文文本。

过程评价关注：

- planning validity
- telemetry
- recovery outcome
- fallback events
- decision trace
- evidence completeness

结果评价关注：

- artifact availability
- artifact validity
- feature count
- geometry readability
- schema validation
- large-area runtime statistics

### 10.4 技术工程实现

主要实现位于 `services/run_report_service.py`：

- `build_run_report_summary()` 构建 JSON summary。
- `render_run_reports()` 输出 summary、中文报告和英文报告。
- `_large_area_runtime_from_events()` 从 audit events 提取 tile count、stitched feature count 和 evidence paths。
- `_artifact_metrics()` 调用 `evaluate_vector_artifact()` 读取产物指标。

`services/artifact_evaluation_service.py` 提供 agentic metrics 和 vector artifact metrics，包括 self-evolution 相关指标：

- `self_evolution_record_written`
- `self_evolution_hint_available`
- `self_evolution_hint_used`
- `self_evolution_policy_adjustment`
- `self_evolution_learning_opportunity_recorded`

### 10.5 功能设计初衷与思想

报告系统服务于 evidence-first 的研究范式。它的设计初衷不是单纯生成漂亮文档，而是让每次融合任务都能成为可复查、可比较、可冻结的实验单元。论文中可以将其作为 RQ3 的重要方法组成。

## 11. 功能 9：错误自动处理，尽量避免人工介入

### 11.1 功能描述

错误自动处理包括运行前拒绝、计划验证修复、执行失败修复、重规划、fallback、checkpoint recovery 和 operator recovery API。目标是在有界任务范围内减少人工介入，使系统面对常见失败时能够自动给出下一步或重新进入运行链。

当前错误处理覆盖：

- unsupported intent guard。
- 参数、输入输出、schema policy 验证。
- source coverage fallback。
- execution failure repair record。
- replan after execution failure。
- checkpoint-based recovery。
- stale run redispatch。
- failed event movement。

### 11.2 能力边界

当前错误处理是 bounded healing，不是无限自愈。边界包括：

- 不支持无限重试，`GEOFUSION_MAX_PLAN_REVISIONS` 限制 plan revision 数。
- 不承诺所有 provider 下载失败可自动恢复。
- 不承诺算法逻辑错误都能自动修复。
- 不承诺生产级 worker crash recovery 和分布式任务一致性。
- 对超出系统能力边界的请求应拒绝，而不是尝试执行。

### 11.3 底层原理

错误处理遵循“先分类，再选择动作”的思想：

1. Preflight 阶段识别 off-domain、trajectory-to-road、unbounded POI 等不支持意图。
2. Validation 阶段尝试修复计划中的参数、类型和 schema 问题。
3. Execution 阶段捕获异常，生成 repair record。
4. Policy engine 判断是 replan 还是 fail。
5. 如果 replan，保留旧 plan，生成 `plan-revision-N.json`，刷新输入并重新验证。
6. 如果 run 卡住，recovery tick 根据 checkpoint 选择 redispatch。

### 11.4 技术工程实现

主要实现包括：

- `services/unsupported_intent_guard.py`
- `agent/validator.py`
- `agent/policy.py`
- `services/run_recovery_service.py`
- `services/run_recovery_executor.py`
- `worker/tasks.py`
- `api/routers/runs_v2.py` 的 operator recovery endpoint
- `services/agent_run_service.py` 中的 replan loop 和 checkpoint 写入

`AgentRunService.execute_run()` 中，执行失败后会构建 failure summary、healing summary、repair records，并根据 policy 判断是否允许 replan。replan 后系统会重新走 validation 和 input acquisition，以避免旧输入与新计划不一致。

### 11.5 功能设计初衷与思想

错误处理功能体现了 FusionAgent 的可靠性哲学：系统不追求“永不失败”，而追求失败可见、原因可解释、修复有边界、过程可审计。论文中应将其表述为 bounded robustness，而不是 autonomous recovery。

## 12. 功能 10：智能体自迭代或自学习

### 12.1 功能描述

当前系统具备 durable learning 和 policy hint 机制。它会在运行完成或失败后记录学习证据，并在后续 pattern selection 中将聚合的 durable learning summary 作为有界分数调整因素。

当前可描述的能力包括：

- 写入 durable learning record。
- 聚合 pattern、algorithm、source 相关历史表现。
- 在 pattern selection candidate evidence 中体现 learning adjustment。
- 在报告中输出 self-evolution 相关指标。
- 在 scenario summary 中暴露 learning opportunity 和 hint usage。

### 12.2 能力边界

这是当前第 10 项与目标之间差距最大的部分。必须明确：

- 当前不是自主自进化。
- 当前不是自动调参系统。
- 当前不会自动修改模型、策略、算法库或 source catalog。
- 当前 learning hint 是有界、可审计、可解释的候选评分调整。
- 论文不应将 C4 作为主贡献，只应作为 supporting claim。

### 12.3 底层原理

当前自学习机制更接近“运行经验的结构化反馈”。其逻辑是：

```text
run outcome
-> execution feedback
-> durable learning record
-> aggregated durable summary
-> retrieval context
-> policy candidate learning_adjustment
-> decision record / report evidence
```

换言之，学习结果不是隐式改变系统，而是以显式证据形式进入下一次决策。这样既能利用历史经验，又能避免不可控的自修改。

### 12.4 技术工程实现

主要实现分布在：

- `services/agent_run_service.py` 中的 `_record_feedback()`
- `kg/repository.py`、`kg/inmemory_repository.py`、`kg/neo4j_repository.py` 的 durable learning record 与 summary 接口
- `agent/retriever.py` 将 durable summaries 放入 planner context
- `agent/policy.py` 对 candidate 做 bounded learning adjustment
- `services/artifact_evaluation_service.py` 与 `services/run_report_service.py` 输出 self-evolution evidence

报告中明确写入边界：

```text
bounded policy hints only; no automatic model, policy, or source catalog mutation
```

### 12.5 功能设计初衷与思想

该功能的初衷是让系统具备“可解释的经验利用”，而不是追求不可控的自动进化。对于论文而言，这一点可以作为可靠智能体设计的一个保守但重要原则：系统可以学习，但学习必须进入审计链路，不能绕开约束和验证。

## 14. 可用于论文的技术方法章节框架

### 14.1 系统总体方法

可以写成：

> 本文提出一种知识图谱约束的地理空间数据融合智能体运行时。该运行时以任务、算法和数据三元关系为核心，将灾害事件转化为场景约束和任务束，并通过受约束规划、工作流验证、数据自动获取、融合执行、错误修复和证据写回，实现建筑物、道路、水系和 POI 多源数据融合。

### 14.2 知识图谱约束规划

重点写：

- 数据类型、算法、任务、参数、数据源和输出 schema policy。
- KG 不只是检索背景，而是执行约束。
- planner 输出需要经过 validator 验证。
- decision record 保留候选、评分、选择理由和证据引用。

### 14.3 共享大范围运行时

重点写：

- 统一抽象 `LargeAreaSlice` 和 `DomainRunner`。
- tile-buffer-owner 机制。
- 输入裁剪、tile 内融合、owner bbox stitch、去重、最终裁剪。
- 统一证据输出。
- 支持 polygon、line、point、building 等 geometry family。

### 14.4 自动数据获取与 source semantic contract

重点写：

- source id 驱动的数据获取。
- 官方源下载、缓存、裁剪、coverage 记录。
- source semantic contract 连接数据源字段、矢量源、栅格源、height policy 与 plan 参数绑定。

### 14.5 反应式修复与重规划

重点写：

- validation repair 与 execution repair 的区别。
- failure taxonomy 与 repair record。
- replan 保留 plan revision。
- checkpoint recovery 和 recovery tick。
- bounded healing 的边界。

### 14.6 证据优先运行机制

重点写：

- 每次 run 生成完整证据。
- 报告不仅展示结果，还展示过程、失败、恢复、fallback、自学习边界。
- 证据契约服务于复现、审计和论文实验冻结。

## 15. 研究背景可写要点

研究背景可以围绕以下逻辑展开：

1. 灾害响应需要快速获取建筑物、道路、水系和 POI 等基础地理数据。
2. 这些数据通常来自 OSM、Microsoft、Overture、HydroSHEDS、GNS/GeoNames 等多源开放数据，存在空间覆盖、格式、语义、精度和更新频率差异。
3. 传统 GIS 处理链依赖人工数据下载、字段清洗、坐标转换、算法选择和结果检查，难以在多场景下快速复用。
4. 大模型带来了自然语言任务理解和流程编排潜力，但开放式 agent 容易出现不可执行计划、工具误用和证据缺失。
5. 因此，需要一种受知识图谱约束、可验证、可修复、可审计的地理空间数据融合智能体运行时。

可以将 FusionAgent 的研究空白定位为：

> 现有 geospatial agent 多关注问答、地图交互或代码生成，而对多源地理数据融合任务中的可执行工作流约束、失败修复、运行证据和大范围融合支持不足。

## 17. 论文贡献可凝练为三点

### 17.1 贡献一：KG 约束的可执行地理融合工作流生成

FusionAgent 将地理融合任务建模为算法、任务、数据和参数之间的受约束组合，通过知识图谱检索和 workflow validation 降低不可执行计划风险。

### 17.2 贡献二：面向多几何族的大范围共享融合运行时

系统将建筑物、道路、水系和 POI 融合统一到 tile、clip、per-tile fusion、stitch、dedupe、evidence 的共享框架下，支持 polygon、line、point 和 building 等不同几何族。

### 17.3 贡献三：证据优先的智能体运行机制

系统将计划、验证、执行、失败、修复、报告和学习提示全部写入可审计证据，支持运行复现、过程评估和论文实验冻结。

## 19. 可直接放入论文的摘要式表述

FusionAgent 面向灾害响应中的多源地理空间数据融合问题，提出了一种知识图谱约束的智能体运行时框架。该框架以“算法 - 任务 - 数据”为可执行核心，将灾害场景转化为任务和数据约束，通过受约束规划、工作流验证、自动数据获取、领域融合执行、反应式修复和证据写回，实现建筑物、道路、水系和有界 POI 数据的自动融合。与开放式工具调用智能体不同，FusionAgent 强调有限任务域内的可执行性、可审计性和可复现性。系统通过共享大范围运行时支持 tile 化裁剪、分块融合、拼接去重和证据输出，并通过 durable learning 机制记录历史运行经验，以有界 policy hint 的形式影响后续规划。实验设计可围绕规划有效性、融合执行成功率、错误恢复能力和证据完整性展开，从而验证知识图谱约束、反应式修复和证据优先机制对地理融合智能体可靠性的作用。

## 20. 结论

截至 2026-05-29，FusionAgent 已经从早期原型推进到一个具备明确研究主线、共享运行时和证据契约的地理空间数据融合智能体。第 2-6 项核心融合能力已经基本闭合，并通过 2026-05-28 的运行时证据冻结得到支撑；第 1、7、8、9 项具备工程可用基础，但仍应按 no-UI 研究运行时而非生产系统表述；第 10 项目前是有界学习提示，而非真正自主自进化。

论文写作上，当前最稳妥的主线不是宣称“通用自主 GIS 智能体”，而是强调：FusionAgent 在有界灾害响应任务中，通过知识图谱约束、共享融合运行时、反应式修复和证据优先机制，实现了可执行、可审计、可复现的多源地理空间数据融合。

## 0529 目标 1、2、5、7-9 差距补齐计划摘要

本轮补齐工作的研究定位不是重新设计融合算法，而是把已有建筑物、道路、水系、POI 与大范围拼接裁剪能力推进到可无人值守运行、可审计、可报告、可恢复的智能体系统状态。核心思路是围绕运行链路增加证据层：调度与 inbox 证明任务可以自动进入系统，source materialization manifest 证明数据可自动下载、缓存、裁剪并记录故障，report quality summary 证明报告能够同时覆盖过程评估和结果评估，recovery hint 证明错误能够被分类并给出自动恢复或人工复核动作。

能力边界需要在论文材料中保持清晰：Target 10 当前是 bounded policy hints only，不声称自主改写知识库；POI 能力是 AOI-bounded OSM + GNS/GeoNames 融合，不声称开放域 POI 实体对齐；自动下载能力受外部数据源可用性影响，因此以 provider attempt、cache_hit、source_mode、fault classification 作为可复现实验证据。

该补齐计划完成后，论文的研究背景、系统方法、工程实现、实验设置和消融实验条件可以同步展开：实验变量包括是否启用自动下载、是否启用高度栅格、是否启用大范围 tile/stitch、是否启用恢复 worker、是否启用质量报告；评价指标包括任务成功率、人工介入次数、下载/缓存命中情况、融合产物有效性、报告完整性和错误恢复有效性。
